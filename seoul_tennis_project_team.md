## RESEARCH: 네이버 예약 테니스장 조사 결과

### 조사 일자: 2026-04-20

### 결론

네이버 예약(booking.naver.com)은 **공식 공개 API를 제공하지 않는다.** 예약 현황 데이터를 가져오려면 **Puppeteer 기반 헤드리스 브라우저 스크래핑**이 현실적으로 유일한 방법이다. 이미 PoC에서 시설 기본정보(시설명, 주소, 코트 목록, 가격, 예약 규칙) 추출에 성공했으며, 날짜별 가용 슬롯 수집은 추가 클릭 자동화로 가능하다.

### 방법론 옵션

| # | 방법 | 실현 가능성 | 설명 |
|---|------|------------|------|
| 1 | **네이버 공식 API** | 불가능 | 네이버 예약 서비스는 외부 개발자용 공개 API를 제공하지 않음. 네이버 개발자센터(developers.naver.com)에 검색/지도/로그인 API는 있으나, 예약(booking) API는 비공개 |
| 2 | **Puppeteer 스크래핑** | **가능 (검증됨)** | 헤드리스 브라우저로 booking.naver.com 페이지 렌더링 후 DOM 파싱. PoC에서 로그인 없이 공개 정보 추출 성공. 날짜 선택 클릭 자동화로 시간 슬롯 데이터도 수집 가능 |
| 3 | **내부 API 역분석** | 부분 가능 | 네이버 예약 페이지가 내부적으로 호출하는 XHR/fetch 엔드포인트를 분석하여 직접 호출. 일부 엔드포인트는 인증 없이 접근 가능할 수 있으나, 언제든 변경될 수 있고 이용약관 위반 가능성 높음 |
| 4 | **RSS/공개 데이터 피드** | 불가능 | 네이버 예약은 RSS나 공개 데이터 피드를 제공하지 않음 |
| 5 | **네이버 플레이스 API** | 불가능 | 네이버 지도/플레이스 API는 장소 검색만 지원하며 예약 현황 데이터는 포함하지 않음 |

### 추천 방법

**Puppeteer 스크래핑 (방법 #2)** 을 추천한다. 이유:

1. **이미 검증됨**: PoC에서 매헌시민의숲테니스장(bizes/210031) 기본 정보 추출 성공
2. **로그인 불필요**: 공개 페이지에서 시설 정보 및 예약 가능 시간 확인 가능
3. **유지보수 가능**: DOM 구조 변경 시 셀렉터만 업데이트하면 됨
4. **Vercel 호환**: Vercel Serverless Function에서 `@sparticuz/chromium` + Puppeteer 조합으로 실행 가능 (단, 실행 시간 제한 주의 — Hobby 플랜 10초, Pro 플랜 60초)

**구현 흐름:**
```
[Cron/수동 트리거] → [Puppeteer로 네이버 예약 페이지 접속]
  → [시설 기본정보 파싱 (시설명, 주소, 코트 목록, 가격)]
  → [날짜 클릭 → 가용 시간 슬롯 파싱]
  → [JSON 데이터로 변환 → 캐싱/DB 저장]
  → [프론트엔드에서 서울 공공 API 데이터와 통합 표시]
```

### 구현 난이도

| 항목 | 난이도 | 비고 |
|------|--------|------|
| 시설 기본정보 수집 | **낮음** | PoC 완료, 바로 구현 가능 |
| 날짜별 가용 슬롯 수집 | **중간** | 날짜 클릭 자동화 + 동적 로딩 대기 필요 |
| Vercel 서버리스 배포 | **중간~높음** | `@sparticuz/chromium` 패키징 필요, 콜드스타트 시간 및 실행 시간 제한 고려 |
| 다중 테니스장 지원 | **중간** | 네이버 예약 biz ID를 수동 관리하거나, 검색으로 자동 수집 |
| 서울 공공 API 데이터와 통합 | **낮음** | 동일 포맷으로 정규화하여 프론트엔드에 전달 |

### 법적/이용약관 이슈

1. **네이버 이용약관**: 네이버 서비스 이용약관은 "자동화된 수단으로 서비스에 접근"하는 것을 금지하고 있음. 스크래핑은 기술적으로 이에 해당할 수 있음
2. **robots.txt**: booking.naver.com의 robots.txt 정책을 준수해야 함
3. **개인정보**: 예약자 개인정보는 수집하지 않으며, 공개된 시설/가격/가용시간 정보만 수집 → 개인정보보호법 이슈 낮음
4. **실질적 리스크**: 소규모 비상업적 프로젝트로서 공개 정보만 수집하므로 실질적 법적 리스크는 낮으나, IP 차단 가능성은 있음
5. **권장 사항**: 요청 빈도를 최소화(1일 1~2회), User-Agent를 명시, robots.txt 준수
